# 전자결재 납부 요청 후속 처리 설계 기록

## 목적

전자결재와 납부 업무는 별도 데이터로 저장한다. 납부 요청은 `PaymentTask`만 생성하며, 전자결재 ID나 결재선에 의존하지 않는다. 화면에서 함께 보여줄 필요가 생기면 조회 계층에서만 조합한다.

## 사용자 흐름

1. 계산 또는 요청 담당자는 `납부 요청`에서 납부 담당자만 최초 선택하고 바로 등록한다. 결재선은 작성하지 않는다. 업무명·금액·마감일·요청 사유·근거는 모두 선택 입력이다.
2. 선택한 납부 담당자는 브라우저 기본값으로 저장되어 이후 요청에 자동 적용된다. 담당자는 필요할 때만 바꾼다.
3. 시스템은 제목을 `납부 요청`으로 시작하게 자동 생성하고, 해당 담당자에게 업무와 알림을 즉시 전달한다.
4. 기한이 비어 있으면 상태는 `기한 설정 필요`다. 납부 담당자는 `납부 업무`에서 실제 납부 기한을 직접 설정한다.
5. 납부 담당자는 납부 후 실제 납부일·금액·메모·납부확인증을 등록한다.

요청자는 납부 담당자가 `요청 확인`을 누르기 전까지 요청 내용과 요청 첨부파일을 수정할 수 있다. 확인 시점부터 요청 원문은 잠기며, 이후에는 담당자의 기한 설정·납부 처리·납부 증빙만 추가된다.

## 경계와 책임

| 영역 | 책임 | 변경 금지 원칙 |
| --- | --- | --- |
| ApprovalRequest | 승인 근거, 결재선, 결재 이력 | 납부 업무와 ID·상태·파일을 공유하지 않는다. |
| PaymentTask | 담당자, 납부 기한, 실제 납부 결과, 증빙 파일 ID | 독립 생성·수정·완료한다. |
| AttachedFile | GridFS 파일 저장과 다운로드 | 직접 납부 요청의 근거는 `PAYMENT_REQUEST`, 납부 확인증은 `PAYMENT_EVIDENCE`로 구분한다. |

## 데이터 계약

납부 요청 생성 API에 다음 계약을 사용한다.

```json
{
  "name": "2026년 1기 부가세",
  "category": "세금",
  "amount": 12500000,
  "due_date": "2026-07-25",
  "assignee_id": "payer01",
  "description": "고지서 기준 납부"
}
```

`assignee_id`만 최초 요청에 필요하며 나머지 필드는 선택이다. 이전 연동 구조에서 남은 `approval_request_id` 인덱스는 시작 시 제거하며, 업무 데이터는 수정하거나 삭제하지 않는다.

## 독립 모듈로 분리하는 기준

다음 중 하나가 현실화되면 `payment_tasks` 컬렉션과 API를 별도 자금·지급 모듈로 이동한다.

- 반복 납부 자동 생성 또는 월별 자금 계획이 필요할 때
- 회계 전표, 은행 이체, 세금 신고 데이터와 양방향 연동할 때
- 담당자별 업무량·연체 지표를 독립 대시보드로 관리할 때

분리 시 현재 API 계약을 어댑터로 유지한다. 요청 첨부와 납부 증빙은 모두 `PaymentTask`의 파일 ID로 참조하므로 파일 저장소를 교체해도 업무 레코드의 핵심 데이터는 유지된다.

## 향후 확장 후보

- 반복 일정 및 공휴일 보정
- 실제 납부금액과 요청금액 차이의 검토 절차

- 지급처·세목·계정과목 등 마스터 데이터 연동

## 확정된 다음 구현 계획: 기한 알림과 완료 기록 검색

### 배경과 목표

현재 납부 업무의 목적은 카카오톡으로 전달받은 PDF를 출력한 뒤 종이로 기한과 완료 여부를 관리하는 과정을 전자화하는 것이다. 복잡한 기업용 자금 승인 시스템보다 다음 두 가지 사용성을 우선한다.

1. 납부할 날짜를 놓치지 않는다.
2. 과거에 납부한 업무를 PDF와 완료 기록을 포함해 빠르게 찾는다.

### 1순위: 기한 알림

완료되지 않았고 `due_date`가 설정된 납부 업무를 대상으로 한국 시간(`Asia/Seoul`) 기준 알림을 생성한다.

| 시점 | 알림 의미 | 기본 동작 |
| --- | --- | --- |
| D-3 | 마감 임박 사전 안내 | 납부 담당자에게 알림 |
| D-1 | 다음 날 납부 안내 | 납부 담당자에게 알림 |
| 당일 | 오늘 납부 필요 | 중요 알림과 사이드바 건수 표시 |
| 기한 초과 | 아직 완료되지 않은 연체 업무 | 완료할 때까지 목록 최상단에 표시 |

#### 알림 규칙

- `COMPLETED` 상태는 알림 대상에서 제외한다.
- 기한이 없는 업무는 날짜 알림 대신 `기한 설정 필요` 상태로 계속 노출한다.
- 같은 업무와 같은 알림 단계는 한 번만 발송되도록 중복 방지 키를 저장한다.
- 담당자가 기한을 변경하면 새 기한을 기준으로 아직 보내지 않은 단계만 다시 계산한다.
- 납부 완료 처리 직후 예약된 알림은 더 이상 발송하지 않는다.
- 초기 구현은 기존 웹소켓·앱 내부 알림을 사용한다. 이메일·메신저 알림은 이후 확장한다.

#### 완료 조건

- D-3, D-1, 당일, 기한 초과 조건이 한국 날짜 기준으로 정확히 계산된다.
- 새로고침하거나 스케줄러가 다시 실행되어도 같은 알림이 중복 생성되지 않는다.
- 완료되거나 기한이 변경된 업무에 이전 일정의 알림이 발송되지 않는다.
- 알림을 누르면 해당 납부 업무를 바로 확인할 수 있다.

### 2순위: 완료 기록 검색

`납부 완료` 목록에 검색·필터 기능을 추가한다. 기본 정렬은 최근 실제 납부일 순으로 한다.

#### 초기 검색 조건

- 업무명, 분류, 요청 사유, 완료 메모의 통합 키워드 검색
- 실제 납부일 기간 검색
- 요청자와 납부 담당자 구분
- 요청 금액 또는 실제 납부 금액 표시
- 요청 첨부파일과 납부 증빙파일 존재 여부

#### 검색 결과 표시

- 업무명과 상태
- 요청자와 납부 담당자
- 납부 기한과 실제 납부일
- 요청 금액과 실제 납부 금액
- 요청 PDF와 납부 증빙파일 개수 및 다운로드
- 완료 메모

#### 구현 원칙

- `내가 요청한 업무`와 `요청 받은 업무` 구분을 검색에서도 유지한다.
- 검색 조건은 프런트에서 전체 목록을 내려받아 처리하지 않고 API 쿼리로 전달한다.
- 결과가 많아질 수 있으므로 페이지네이션을 적용한다.
- 검색 성능을 위해 `status`, `requester_id`, `assignee_id`, `paid_at`, `due_date` 조합 인덱스를 검토한다.
- 완료 기록의 수정이 허용되더라도 최초 완료 시각과 마지막 수정 시각은 구분해 보존한다.

#### 완료 조건

- 날짜 범위와 키워드를 함께 사용해 원하는 완료 업무를 찾을 수 있다.
- 요청자 화면과 납부 담당자 화면에서 권한에 맞는 기록만 조회된다.
- 검색 결과에서 원본 PDF와 납부 증빙을 바로 확인하거나 다운로드할 수 있다.
- 결과가 없는 경우와 로딩 중인 경우를 명확하게 표시한다.

### 구현 순서

1. 알림 중복 방지를 위한 데이터 구조와 스케줄러 작업 추가
2. 기한별 알림 생성 및 알림 클릭 이동 구현
3. 완료 기록 검색 API와 인덱스 추가
4. 납부 완료 검색 화면과 페이지네이션 구현
5. 시간대 경계, 기한 변경, 완료 처리, 권한별 검색 테스트
