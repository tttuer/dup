# 전자결재 납부 요청 후속 처리 설계 기록

## 목적

전자결재와 납부 업무는 별도 데이터로 저장한다. 납부 요청은 `PaymentTask`만 생성하며, 전자결재 ID나 결재선에 의존하지 않는다. 화면에서 함께 보여줄 필요가 생기면 조회 계층에서만 조합한다.

## 사용자 흐름

1. 계산 또는 요청 담당자는 `납부 요청`에서 납부 담당자만 최초 선택하고 바로 등록한다. 결재선은 작성하지 않는다. 업무명·금액·마감일·요청 사유·근거는 모두 선택 입력이다.
2. 선택한 납부 담당자는 브라우저 기본값으로 저장되어 이후 요청에 자동 적용된다. 담당자는 필요할 때만 바꾼다.
3. 시스템은 제목을 `납부 요청`으로 시작하게 자동 생성하고, 해당 담당자에게 업무와 알림을 즉시 전달한다.
4. 기한이 비어 있으면 상태는 `기한 설정 필요`다. 납부 담당자는 `납부 업무`에서 실제 납부 기한을 직접 설정한다.
5. 납부 담당자는 납부 후 실제 납부일·금액·메모·납부확인증을 등록한다.

요청자는 납부 담당자가 `요청 확인`을 누르기 전까지 요청 내용과 요청 첨부파일을 수정할 수 있다. 확인 시점부터 요청 원문은 잠기며, 이후에는 담당자의 기한 설정·납부 처리·납부 증빙만 추가된다.

## 경계와 책임

| 영역 | 책임 | 변경 금지 원칙 |
| --- | --- | --- |
| ApprovalRequest | 승인 근거, 결재선, 결재 이력 | 납부 업무와 ID·상태·파일을 공유하지 않는다. |
| PaymentTask | 담당자, 납부 기한, 실제 납부 결과, 증빙 파일 ID | 독립 생성·수정·완료한다. |
| AttachedFile | GridFS 파일 저장과 다운로드 | 직접 납부 요청의 근거는 `PAYMENT_REQUEST`, 납부 확인증은 `PAYMENT_EVIDENCE`로 구분한다. |

## 데이터 계약

납부 요청 생성 API에 다음 계약을 사용한다.

```json
{
  "name": "2026년 1기 부가세",
  "category": "세금",
  "amount": 12500000,
  "due_date": "2026-07-25",
  "assignee_id": "payer01",
  "description": "고지서 기준 납부"
}
```

`assignee_id`만 최초 요청에 필요하며 나머지 필드는 선택이다. 이전 연동 구조에서 남은 `approval_request_id` 인덱스는 시작 시 제거하며, 업무 데이터는 수정하거나 삭제하지 않는다.

## 독립 모듈로 분리하는 기준

다음 중 하나가 현실화되면 `payment_tasks` 컬렉션과 API를 별도 자금·지급 모듈로 이동한다.

- 반복 납부 자동 생성 또는 월별 자금 계획이 필요할 때
- 회계 전표, 은행 이체, 세금 신고 데이터와 양방향 연동할 때
- 담당자별 업무량·연체 지표를 독립 대시보드로 관리할 때

분리 시 현재 API 계약을 어댑터로 유지한다. 요청 첨부와 납부 증빙은 모두 `PaymentTask`의 파일 ID로 참조하므로 파일 저장소를 교체해도 업무 레코드의 핵심 데이터는 유지된다.
